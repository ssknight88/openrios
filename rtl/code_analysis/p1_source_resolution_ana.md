# p1_source_resolution.sv 深度解析 (寄存器重命名)

这个模块实现了乱序执行中最关键的一环：**寄存器重命名 (Register Renaming)** 和 **操作数源解析 (Source Resolution)**。它决定了一条即将进入 ISQ 的指令，究竟是应该去读现成的物理寄存器，还是要挂起去监听别人的结果。

---

## 1. 核心职责
将架构寄存器索引 (Architecture Register, 例如 `x5`) 转换为物理状态 (Physical State, 例如 ARF 里的真实数据，或者是正在计算它的指令的 ROB Tag)。

## 2. 与外部表单的交互
模块连接了两个最重要的数据结构：
*   **ARF (架构寄存器堆)**：存放已经安全退休的真实数据。
*   **DST_REG (目标寄存器状态表)**：记录每个架构寄存器当前的“期货状态”。
    *   `busy`: 为 1 表示该寄存器的新值正在被某条较老的指令计算。
    *   `tag`: 如果 busy 为 1，这里存的就是那条老指令在 ROB 中的 Tag。

## 3. 核心机制：三步走解析法

对于指令的每一个源操作数（`rs1`, `rs2`, `rs3`），模块都执行以下组合逻辑流程：

### 步骤 A：查表 (Read DST_REG)
*   如果 `DST_REG` 报告这个操作数 `busy`：
    *   说明数据还没算出来。
    *   **动作**：`ready = 0`, `tag = dst_rtag` (记录在案，准备去 ISQ 里监听这个 Tag)。
*   如果 `DST_REG` 报告 `!busy`：
    *   说明数据已经是最终版本了。
    *   **动作**：`ready = 1`, `data = arf_rdata` (直接把 64 位硬核数据读出来带走)。

### 步骤 B：同拍前递覆盖 (Same-Cycle Rename Overlay)
这是超标量双发处理器中极度精妙的一笔！
*   **场景**：如果当拍双发的两条指令是 `ADD x5, x1, x2` (Slot 0) 和 `SUB x6, x5, x3` (Slot 1)。
*   **问题**：在查 `DST_REG` 时，Slot 0 还没来得及把 `x5` 标成 busy！Slot 1 可能会错误地读到以前过期的 `x5` 数据。
*   **解决**：模块强制实施一个组合逻辑旁路。如果 Slot 1 发现自己需要的 `rs` 恰好等于 Slot 0 正在写的 `rd`，它会直接短路：
    *   `ready = 0`
    *   `tag = slot0_alloc_tag` (直接监听 Slot 0 刚拿到的这块新鲜出炉的 Tag)。

### 步骤 C：P4 极晚期前递 (Commit Bypass)
*   **场景**：如果 `DST_REG` 说 busy 且给出 Tag=A。但在这一模一样的时钟周期，Tag A 恰好正在 P4 Commit（即写 ARF）！
*   **解决**：如果不处理，这条新指令会拿着 Tag A 进 ISQ 傻等，但 Tag A 永远不会在 Bypass 总线上出现了（它已经退休了）。所以，模块引入了 `commit_payload` 检查：如果监听的 Tag 恰好正在当拍提交，则当场把提交的数据“截胡”，并标记 `ready = 1`。

## 4. 总结
这个模块是“虚拟时序（程序顺序）”与“物理时序（数据准备顺序）”的交汇点。经过这层洗礼，无论软件怎么折腾寄存器，硬件都能精准地用 Tag 把真实的数据依赖链（Data Dependency Graph）构建出来。
